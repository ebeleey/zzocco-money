## 깃허브 협업
### 커밋 규칙
- **커밋 메시지 형식**
  - **타입**: `feat`(새로운 기능), `fix`(버그 수정), `docs`(문서 수정), `chore`(기타 변경)
  - **예시**: `feat: Add financial style analysis feature`
- **커밋 메시지 본문**
  - 변경 사항에 대해 자세한 설명 포함
- **커밋 빈도**
  - 작은 단위로 자주 커밋
### 브랜치 전략- Github Flow
- **메인 브랜치**: `main`
  - 항상 배포 가능한 상태를 유지
  - 모든 Pull Request는 이 브랜치로 머지
- **기능 브랜치**: `feature/기능명`
  - 새로운 기능 추가 시 생성
  - 작업 완료 후 `main`으로 Pull Request 작성 및 리뷰 진행
### 작업 규칙
1. `main` 브랜치는 직접 수정하지 않습니다.
2. 새로운 작업은 항상 `feature/기능명` 브랜치에서 시작합니다.
3. Pull Request 작성 시, 변경 사항과 테스트 내용을 상세히 기재합니다.
4. 모든 머지는 코드 리뷰를 통해 진행합니다.
